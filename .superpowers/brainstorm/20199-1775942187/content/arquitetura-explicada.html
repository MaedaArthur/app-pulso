Como o app funciona por dentro

Cada camada tem uma função específica. Você escreve menos código porque cada lib resolve um problema bem definido.

Camada 1 — O que o usuário vê
Telas (React Components)
São os arquivos .tsx — o que aparece na tela. A home, a tela de entradas, o formulário de CSV. Eles não sabem nada sobre banco de dados. Só recebem dados e mostram.
Analogia: o garçom que traz o prato. Ele não cozinha, não faz o pedido para o fornecedor — só entrega o que recebe.
Home.tsx Entradas.tsx Gastos.tsx Onboarding.tsx
↕chama hooks para pedir dados
Camada 2 — Pontes de dados (Custom Hooks)
useEntradas · useGastos · useSaldo
Cada hook é responsável por um domínio. useGastos() sabe buscar os gastos do mês atual, filtrar por categoria, somar. A tela só chama o hook e recebe o resultado — não precisa saber de onde vem.
Analogia: o garçom vai até a cozinha para buscar o prato. A cozinha (TanStack Query) decide se vai realmente cozinhar ou se já tem o prato pronto.
useEntradas.ts useGastos.ts useSaldo.ts useOnboarding.ts
↕hooks usam TanStack Query para buscar/salvar
Camada 3 — Memória inteligente (TanStack Query)
Cache + sincronização automática
O TanStack Query é a "memória de curto prazo" do app. Quando você pede os gastos do mês, ele verifica: "já tenho isso em memória e é recente?" Se sim, responde na hora — sem ir ao banco. Se não, vai buscar no Supabase, guarda o resultado, e na próxima vez responde instantâneo.

Também cuida de estados como "está carregando" e "deu erro" — sem você escrever isso manualmente.
Cache em palavras simples: você pergunta ao Google Maps o caminho para casa. Ele te responde em 0,1s porque já calculou antes. Se a rua fecha, ele invalida o cache e recalcula. É exatamente isso — guardar respostas para não precisar recalcular toda hora.
↙busca dados do banco
↘aplica lógica financeira
Camada 4A — Banco de dados
Supabase
O banco de dados real. Guarda entradas, gastos, perfil do usuário, eventos de analytics. Também cuida do login (auth).
Analogia: o depósito do restaurante. Tudo que existe fica aqui.
tabela: entradas tabela: gastos tabela: perfil tabela: events
Camada 4B — Cálculos puros
finance.ts
Funções matemáticas puras. Recebe números, devolve números. Não sabe o que é React, não sabe o que é banco de dados. Só calcula.
Vantagem: é testável de forma isolada. Se a fórmula do saldo estiver errada, você descobre no teste antes de ver na tela.
calcularSaldo() estadoDoMes() ritmoDeMes()

O que acontece quando você abre o app?

1
Tela (React)
Home.tsx renderiza
A tela aparece. Imediatamente chama useSaldo() e useGastos().
2
TanStack Query (cache)
Tem cache? → Mostra na hora. Não tem? → Mostra skeleton
Se você já abriu o app nos últimos 5 min, os dados aparecem instantaneamente do cache. Se é a primeira vez, mostra loading enquanto vai buscar.
3
Supabase
Busca entradas e gastos do mês atual do banco
Uma query: "me dê todos os gastos de Arthur em abril de 2026". O Supabase responde com a lista.
4

Calcula saldo, estado do mês, ritmo
calcularSaldo(dinheiro_guardado, gastos, meta) → R$ 847. estadoDoMes(847, media_renda) → "verde".
5
Tela (React)
Home atualiza com os dados reais
O TanStack Query avisa a tela que os dados chegaram. A tela re-renderiza mostrando "Verde ✓ R$ 847". O cache guarda esse resultado.

⚡ Cache — por que isso importa?

Sem cache, cada vez que você muda de aba o app vai ao banco de dados buscar os mesmos dados de novo. Com cache, ele guarda o resultado e reutiliza. A experiência fica instantânea — sem loading toda hora.

Sem cache

  • Abre o app → busca no banco (500ms)
  • Vai para Entradas → busca de novo (500ms)
  • Volta para Home → busca de novo (500ms)
  • Sem internet → tela vazia

Com cache (TanStack Query)

  • Abre o app → busca no banco (500ms)
  • Vai para Entradas → instante (memória)
  • Volta para Home → instante (memória)
  • Sem internet → mostra último dado
